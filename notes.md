# Исследование безопасности SberBulb A60 E27

## Устройство

Умная лампочка от Сбера. Работает по WiFi (2.4 ГГц), управляется через приложение SberDevices.
При первом включении поднимает собственную точку доступа для настройки — телефон подключается к ней и передаёт параметры домашней сети.
После настройки лампочка подключается к домашнему роутеру и дальше общается с облаком Сбера.

Железо свистка для перехвата: Ralink RT5370, интерфейс wlan1.

---

## Анализ дампов (что снималось заранее)

Захваченные файлы:
- `sber_pairing-01.cap` — запись момента сопряжения
- `sber_pairing2-02.cap` — ещё одна запись с паринга
- `sber_attack-01.cap` — запись во время тестовой deauth-атаки
- `wire_scan(udp+dora...).pcapng` — трафик уже подключённой лампочки

Смотрелось всё через tshark (консольный wireshark).

### Что нашли в beacon-фрейме лампочки

Лампочка в режиме паринга поднимает AP с именем `SberBulb D05F`.
В beacon-кадре видно:

```
RSN Information:
  Group Cipher: AES (CCM)
  AKM: PSK
  Management Frame Protection Required: Not required
  Management Frame Protection Capable: Not capable
```

Последние две строчки — ключевое. Это значит что лампочка **не поддерживает MFP (802.11w)**.
MFP — это защита управляющих фреймов от подделки. Без неё фреймы деаутентификации можно отправлять от имени кого угодно, и устройство их примет.

### Трафик после подключения к домашней сети

Лампочка получала IP 192.168.0.123, после этого:
- DNS запрос к 8.8.8.8: `odm.iot.sberdevices.ru`
- TCP подключение к 109.238.88.134, порт 8883
- Это MQTT over TLS — стандартный протокол для IoT-устройств

TLS хендшейк показал интересное. Лампочка предлагает только два шифра:
```
TLS_RSA_WITH_AES_256_CBC_SHA256
TLS_RSA_WITH_AES_128_CBC_SHA256
```

Оба без Perfect Forward Secrecy. Ключ сессии выводится из RSA-ключа сервера.
Если приватный ключ сервера утечёт — весь исторический трафик расшифруем.

Сертификат сервера самоподписанный (issuer == subject), CN = `odm.iot.sberdevices.ru`.

### Полный 4-way хендшейк в attack-дампе

В `sber_attack-01.cap` airodump-ng нашёл один полный WPA2 хендшейк для `TP-Link_5E56`.
Это домашний роутер — лампочка переподключалась к нему, и хендшейк был пойман.
По идее можно попробовать взломать пароль через словарь, но пароль уже был известен.

## Утилиты которые использовались

**airmon-ng** — переводит WiFi-адаптер в monitor mode. В обычном режиме адаптер видит только те пакеты, которые адресованы ему. В monitor mode видит весь эфир.

**airodump-ng** — пассивный сканер. Слушает эфир в monitor mode и записывает всё что видит: AP, клиентов, уровни сигнала, каналы. Пишет в csv и cap.

**aireplay-ng** — активный инструмент. Флаг `-0` отправляет deauth фреймы — говорит клиенту "ты отключён" от имени AP. Работает без MFP.

**hostapd** — демон для поднятия точки доступа на Linux. Используется для Evil Twin — поднимаем AP с теми же параметрами что у реального роутера.

**dnsmasq** — лёгкий DNS + DHCP сервер. Раздаёт IP-адреса подключившимся клиентам и отвечает на DNS запросы. В нашем случае — перенаправляет `odm.iot.sberdevices.ru` на наш IP.

**iptables** — брандмауэр Linux. Используется для перенаправления трафика. Конкретно: пакеты на порт 8883 (MQTT) перенаправляются на наш прокси.

**openssl** — генерация сертификатов. Делаем самоподписанный сертификат с тем же CN что у реального сервера Сбера — лампочка его получит вместо настоящего.

## Evil Twin атака — суть

Идея: создать точку доступа, неотличимую от домашнего роутера лампочки.
Лампочка не проверяет подлинность AP — ей важны только SSID, BSSID и пароль.

Что мы клонировали:
- SSID: `TP-Link_5E56`
- BSSID: `D8:0D:17:7F:5E:56`
- канал: 10
- WPA2 PSK пароль: `20272791` (из нашей же сети)

Дальше через deauth-атаку лампочку отключили от реального AP — она начала сканировать и нашла "знакомую" сеть с нашим сигналом.

Параллельно:
- dnsmasq раздаёт IP из подсети 10.0.13.x
- DNS отвечает на `odm.iot.sberdevices.ru` нашим IP 10.0.13.1
- iptables перенаправляет порт 8883 → 18883 (наш прокси)
- python-скрипт слушает на 18883, пытается перехватить TLS

## Результаты

**Deauth** — работает. Лампочка отключалась от реального AP без проблем. Причина: нет MFP.

**Подключение к Evil Twin** — работает. Из hostapd.log:
```
wlan1: AP-STA-CONNECTED fe:3c:d7:a3:d0:5f
wlan1: EAPOL-4WAY-HS-COMPLETED fe:3c:d7:a3:d0:5f
```
Лампочка прошла WPA2 аутентификацию с нашим AP несколько раз подряд.

**DHCP** — работает. Из файла `/var/lib/misc/dnsmasq.leases`:
```
fe:3c:d7:a3:d0:5f  10.0.13.109
```
Лампочка получила IP от нашего dnsmasq.

**MQTT MITM** — не подтверждён. Прокси подключений не зафиксировал.
Скорее всего лампочка проверяет TLS сертификат (certificate pinning или проверка цепочки CA).
Соединение устанавливалось на TCP уровне (мы видим что лампочка подключается), но TLS хендшейк падал до того как прокси мог что-то залогировать — устройство сразу разрывало соединение.

Это само по себе результат — значит хотя бы один уровень защиты (TLS) в устройстве есть.

## Итоговые уязвимости

1. **Нет MFP** — deauth атака работает без ограничений. Лампочку можно в любой момент отключить от сети. DoS по факту.

2. **Нет верификации AP** — лампочка подключается к любому AP с правильными SSID+пароль. Нет никакой проверки "а тот ли это роутер". Это классическая уязвимость WPA2-Personal — протокол не аутентифицирует AP, только клиента.

3. **Нет PFS в TLS** — шифры без Forward Secrecy. Если ключ сервера скомпрометирован — весь сохранённый трафик читаем.

4. **Самоподписанный сертификат сервера** — сервер Сбера использует self-signed cert. Нет публичного CA. Это затрудняет верификацию клиентом.

5. **WPA2-PSK (не Enterprise)** — в корпоративных сетях каждое устройство имеет свои credentials. В PSK — один пароль на всех. Если пароль домашней сети стал известен атакующему — Evil Twin реализуем.
